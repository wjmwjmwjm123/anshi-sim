# 《安史之乱：中唐续命》详细项目规划书

> 文档版本：0.1（立项首稿）  
> 日期：2026-07-11  
> 项目起点：天宝十五载六月，潼关战役前夕  
> 目标平台：Windows PC，优先 Steam；桌面单机，联网调用 LLM，可降级离线运行  
> 文档用途：统一产品范围、玩法设计、目标架构、研发顺序和验收口径

---

## 0. 立项结论

本作不应是《明末力挽狂澜》的唐代换皮，也不应继续沿用“LLM 先写整篇结果、多个 extractor 再反复阅读并修改数值”的核心结算方式。

建议把项目定义为一款 **史料驱动、角色对话增强、确定性规则主导的短周期历史危机模拟游戏**。玩家从 756 年六月潼关战役前夕开始，经历潼关、马嵬、灵武、两京收复、相州、史思明复叛直至 763 年乱事终结。游戏关注的不是简单“打赢安史之乱”，而是用多种代价维持王朝、军队、财政和权力秩序，并决定战后大唐会留下怎样的国家。

目标方案的四项核心决策：

1. **程序规则拥有最终裁决权。** 战斗、补给、军令执行、忠诚、财政、事件门槛和结局评分由可测试的规则计算，LLM 不直接写数据库数值。
2. **多 AI 是并行的专业角色，不是多个模型轮流聊天。** 军事、情报、朝局、人物表达和叙事分别负责有限任务，共享同一份只读快照，输出严格结构化建议。
3. **玩家先看到结果，叙事后补。** 确定性结算完成后立即更新盘面；战报、露布和人物反应异步流式生成，不阻塞下一步操作。
4. **首发只做唐廷主视角。** 叛军视角、多周目继承、完整科技树和大规模战术微操不进入 MVP，避免内容与系统同时失控。

## 1. 项目定位

### 1.1 一句话定义

玩家扮演危局中的唐朝最高决策者，在错误情报、失控军权、断裂补给和朝廷内斗之间下达有限命令，争取平乱，同时避免胜利本身制造更严重的藩镇时代。

### 1.2 核心卖点

- **潼关不是一场数值战，而是一场决策链崩溃。** 玩家必须处理假情报、监军干预、将相互疑、主帅病情、部队构成和地形风险。
- **大臣不是菜单。** 关键人物有立场、信息边界、关系和私利，同一问题会给出相互冲突但各自合理的意见。
- **战役胜利不等于国家胜利。** 借兵回纥、招降叛将、扩张节度使权力都能救急，也会变成终局债务。
- **史料可追溯。** 史实锚点、合理推演和虚构分支分级标注，玩家可查看简明出处和史料争议。
- **短战役、高密度。** 聚焦 756-763 年，不做跨千年的泛历史沙盘。

### 1.3 目标用户

| 用户 | 主要需求 | 本作提供 |
|---|---|---|
| 历史策略玩家 | 决策有后果、系统能解释历史 | 可验证规则、史料锚点、结构性困境 |
| 中国史爱好者 | 人物与事件尊重史实 | 史料注释、争议说明、历史分支标记 |
| 叙事玩家 | 人物鲜活、选择有情绪重量 | 多角色朝议、私下召对、动态战报 |
| 轻度玩家 | 不被复杂数值劝退 | 剧情难度、推荐方案、自动军务、失败续玩 |

### 1.4 明确不做

- 不做《全面战争》式实时战术操控。
- 不让 LLM 自由决定兵力增减、城池归属或人物生死。
- 不在首发同时制作唐廷与叛军两条完整战役。
- 不以 68 名人物全部常驻 Agent 作为“多 AI”卖点。
- 不复制参考项目的巨型 `web_app.py`、全局内容注入和四次重复抽取链。
- 不把史实结果强制写死；历史事件是压力与候选，不是无条件播放的过场。

## 2. 调研与资产基线

### 2.1 可继承内容

`E:\AGENT\二轮调研` 已形成可用于策划输入的内容库：

| 资产 | 当前规模 | 规划用途 |
|---|---:|---|
| 世界状态 | 4 个全局指标、3 个开局危机、7 个帝国遗产 | 开局盘面与长期结构性修正 |
| 军队与势力 | 研究目标为 21 支武装、唐/叛/外部三类 | 战区、军队、统帅和控制关系 |
| 人物 | README 声称 68 人，主文件统计仍显示 44 人 | 核心人物池、后期登场与关系网 |
| 地区 | 22 地区、16 条拓扑边、灵宝战场细节 | 战略地图、补给和事件空间 |
| 事件 | 15 个五幕史实事件 | 主线锚点与分支模板 |
| 系统 | 10 个机制草案 | 规则设计输入，不直接照搬公式 |
| 史料与演出 | 10 条主要来源、20 组人物语料、20 个 CG 脚本 | 注释、角色声音和关键演出 |

### 2.2 数据进入制作前必须完成的审计

1. **人物数量冲突。** `D3_Characters.json` 的 `summary_statistics.total_characters` 为 44，而 README 与 patch 合并目标为 68。必须重新执行合并、按 `char_id` 去重并生成校验报告。
2. **地图连通性不足。** 22 个地区只有 16 条边，默认不能形成完整战略网络。需明确“不可直接通行”与“数据漏边”，补充水路、驿道和跨区补给边。
3. **兵力口径分离。** 同时保存账面兵力、可战兵力、编制质量和当前可集中兵力，不能只留一个 `manpower`。
4. **史料质量分级。** 正史、编年史、现代论文、百科和网络文章必须分级。百科可作线索，不作为关键数值的唯一依据。
5. **事件时间校准。** 开局确定为 756 年六月后，755 年起兵和封常清、高仙芝事件只能作为序章、回忆或历史背景，不能伪装成玩家可改变的当期事件。
6. **公式去伪精确。** 调研中的概率与系数多为设计草案，需经过模拟和测试校准，不能以“史实锚定”名义直接定值。

### 2.3 参考项目可吸收与应重写的部分

可吸收：

- 朝议、拟旨、结算、决策点构成的可理解循环。
- 内容数据与引擎分离的方向。
- 章节记忆、人物记忆和存档隔离的产品经验。
- 前端地图、人物召见、奏报和结局史册等交互概念。

应重写：

- LLM 叙事后再多次抽取数值的结算链。
- 规则散落在数据库层、Agent 工具和清洗器中的状态修改方式。
- 单例全局内容、进程内 pending decision、巨型 API 文件。
- 每回合多个高级模型串行调用的成本与时延结构。

## 3. 游戏时间与战役结构

### 3.1 采用“事件时钟”，不固定一回合一月

潼关失守、长安出逃、马嵬兵变发生在极短时间内，而整场安史之乱持续八年。固定月回合会让开局失真，固定日回合又会产生大量垃圾时间。

建议使用三种时间尺度，由剧情和行动自动切换：

| 时间尺度 | 单回合跨度 | 使用场景 |
|---|---:|---|
| 危机回合 | 1-3 日 | 潼关催战、出逃、马嵬、政权分裂等高密度节点 |
| 战役回合 | 10-15 日 | 两京反攻、相州会战、围城与补给争夺 |
| 战略回合 | 1-3 月 | 战线僵持、财政恢复、藩镇整顿和外交 |

完整战役目标控制在 **36-48 个有效回合、首通 6-10 小时**。玩家可在无重大事件时自动推进；自动推进仍执行确定性规则，但只展示异常和关键变化。

### 3.2 五幕结构

| 幕 | 时间 | 核心问题 | 典型结局门槛 |
|---|---|---|---|
| 第一幕：潼关危局 | 756.06 | 守还是战、信谁、如何保住指挥链 | 潼关守住、主动撤退、灵宝惨败 |
| 第二幕：皇统裂变 | 756.06-756.12 | 长安、马嵬、玄宗与太子权力交接 | 成都行在、灵武朝廷、双中心内耗 |
| 第三幕：收复两京 | 757 | 回纥条件、睢阳牺牲、长安洛阳次序 | 收复速度、民生代价、外援债务 |
| 第四幕：胜而未定 | 758-760 | 九节度协调、降将处置、史思明复叛 | 联军崩解、叛军重组、中央威望 |
| 第五幕：平乱之后 | 761-763 | 史朝义末路、河朔安排、吐蕃压力 | 中兴、脆弱和平、藩镇割据、王朝再崩 |

### 3.3 开局垂直切片

首个可玩的完整切片只做 756 年六月：

1. 收到陕郡敌弱情报。
2. 分别召见杨国忠、哥舒翰、边令诚、高力士或太子代表。
3. 玩家决定是否复核情报、调换监军、坚守、试探或出关。
4. 程序结算军令执行、侦察结果、伏击风险和军队状态。
5. 触发灵宝战役或替代分支。
6. 根据结果进入长安守备、撤离或政变危机。
7. 形成一份可追溯的“为何如此”战报。

这一切片必须先证明三件事：选择真的改变局势、失败可以被理解、单回合等待可接受。

## 4. 核心循环

```text
阅读盘面与异议
    ↓
召见人物 / 查询军情 / 复核史料线索
    ↓
编组本回合命令（军令、诏令、任命、外交、密令）
    ↓
预览执行风险与信息可信度
    ↓
确定性模拟内核结算
    ↓
立即更新地图、军队、财政、忠诚与事件
    ↓
异步生成露布、朝议反应和人物记忆
    ↓
处理少量真正需要玩家裁决的新危机
```

### 4.1 玩家每回合的稀缺资源

- **皇帝注意力**：限制召见、复核、亲自干预的次数。
- **朝廷执行力**：决定一回合可同时下达多少复杂命令。
- **政治资本**：用于压制派系、越过程序、撤换大将。
- **军费与粮秣**：用于调兵、犒赏、征募和外援。
- **可信情报**：不是通用货币，而是对具体对象、地区和时间有效的证据。

限制资源的目的不是增加操作，而是迫使玩家决定“这回合真正关心什么”。

### 4.2 命令必须具备的字段

所有命令都转成统一、可验证的结构：

```json
{
  "type": "hold_position",
  "issuer": "xuanzong",
  "target": "tongguan-army",
  "objective": "守潼关并等待河北战果",
  "priority": 90,
  "resources": {"grain": 12000, "gold": 3000},
  "constraints": ["不得出关追击", "斥候每日三报"],
  "expires_at": "756-06-15"
}
```

LLM 可以帮助玩家把自然语言整理成命令草案，但只有通过 schema、权限、资源和冲突检查的命令才能进入模拟器。

## 5. 核心系统规划

### 5.1 情报与认知

情报不是简单的“准确率”数值，而是带来源和置信度的对象：

- 事实对象：敌军位置、兵力区间、主将、补给、地形布置。
- 来源对象：斥候、地方官、监军、降人、商旅、密探。
- 误差对象：过期、夸大、隐瞒、伪造、来源串联。
- 玩家看到的是多份互相矛盾的报告，而不是系统直接标注真假。
- 复核、交叉验证和直接召见会提升某条事实的置信度。

程序负责生成和保存真实世界状态；AI 只负责按人物知识边界表达报告。这样既能制造迷雾，也能在复盘时解释假情报从何而来。

### 5.2 军令执行与忠诚

军队不是点一下就移动。每条军令经过以下链条：

```text
合法性 → 送达 → 理解 → 接受 → 准备 → 执行 → 回报
```

执行率由统帅忠诚、对皇帝/宰相/监军的信任、军令清晰度、距离、补给、军心和个人风险共同决定。低执行率不一定等于叛变，也可能表现为拖延、保守执行、虚报完成或擅自变更。

必须避免“忠诚低于 20 就随机叛变”的单阈值设计。叛变应需要动机、能力、机会和替代归属同时成立。

### 5.3 战役与战斗

战斗使用确定性公式加受控随机种子，核心输入包括：

- 可战兵力与兵种构成，而非账面兵力。
- 士气、训练、装备、疲劳、伤病和欠饷。
- 主帅与分军指挥能力、指挥冲突。
- 地形宽度、伏击区、退路容量、天气和能见度。
- 情报优势、先手、阵型和行动目标。
- 补给状态与撤退计划。

输出不仅是胜负，还包括伤亡区间、溃散、被俘、退路、军械损失、控制权变化和对其他战区的士气冲击。

灵宝战役使用专用地图规则，但不能写死结果。历史惨败是开局默认条件组合的自然产物：错误情报、狭道、指挥不一、主帅病重、退路超载和伏击共同作用。

### 5.4 补给与交通

- 地区构成有向交通图，边包含距离、道路、水路、季节、容量和风险。
- 军队补给从粮仓沿控制路径运输，存在损耗、截断和拥堵。
- 城池可有存粮而前线仍缺粮，避免“全国粮食一个池子”。
- 征发民夫能提升短期运输，却会降低地区生产、民心和治安。
- 睢阳、南阳等围城通过同一规则运行，特殊事件只负责补充人物与伦理选择。

### 5.5 朝局、派系与信任网络

派系不是固定阵营标签，而是围绕具体议题形成的关系网络。系统至少维护：

- 皇帝与太子、宰相、宦官、边将、节度使之间的双向信任。
- 人物之间的恩怨、师承、部曲、婚姻和共同经历。
- 派系对议题的立场，例如守潼关、借回纥、招降史思明、战后削藩。
- 人物掌握的信息与愿意公开的信息分离。

朝议的价值是暴露冲突和盲区，不是让多个 Agent 输出五段相似建议。

### 5.6 叛军联盟

叛军按安禄山直属、史思明部、地方降军和族群武装分别建模。联盟稳定性来自利益分配、胜败、首领威望、补给和继承问题。

玩家可以离间、招降和赦免，但每次接受带兵投降都留下战后自治债务。史实中的弑主、复叛和河朔三镇应由联盟与招降规则共同产生，而非仅靠定时事件。

### 5.7 外交与外援债务

回纥援军不是一次性“花钱加兵力”：

- 谈判明确和亲、岁币、战利品、行动范围和撤军条件。
- 借兵形成可追踪的债务与承诺。
- 外援部队有独立目标、补给和纪律，可能拒令或劫掠。
- 吐蕃依据唐朝西线空虚、威望和边区兵力动态调整侵略倾向。
- 外交结果进入终局评价，不允许“借完即忘”。

### 5.8 财政与民生

MVP 不需要复制完整明代财政表。首发只保留能形成决策的科目：

- 中央现银、粮储、江淮转运、军费承诺、拖欠军饷。
- 地区人口、生产、治安、流民和税负。
- 临时征发、盐税改革、借贷、减免和军队就地筹饷。
- 战争破坏与恢复速度。

所有财政流水由程序记账。叙事只能解释交易，不能新造一笔没有来源的银粮。

### 5.9 皇统与操控角色

开局玩家扮演玄宗，但“玩家”代表的是本局最高决策意志，不必因灵武即位立即结束游戏：

- 若太子北上即位，玩家选择继续扮演玄宗、转为肃宗或进入有限双中心模式。
- 双中心模式下，两套朝廷各有资源、人物和命令权限，不能无成本共享情报。
- 君主更替保留国家历史，但改变可用人物、政治资本和胜利条件。

MVP 可先实现玄宗转肃宗的单向切换；完整双中心作为 Alpha 后功能。

### 5.10 结局与评价

结局至少由六个维度共同决定：

1. 皇统连续性。
2. 两京与核心领土控制。
3. 叛军组织是否瓦解。
4. 中央军占比与藩镇自治债务。
5. 财政、人口和民生损失。
6. 回纥、吐蕃与边疆代价。

建议结局不是简单胜负，而是“中兴但失边”“快速平乱、藩镇坐大”“两京复得、民生崩坏”“潼关守成、中央改革”“王朝分裂”等组合评价。

## 6. 多 AI 目标架构

### 6.1 设计原则

- Agent 只能读取经过裁剪的世界快照，不能直接操作数据库。
- 每个 Agent 只负责一种产物，并有固定 schema、超时和降级方案。
- 同一阶段无依赖的任务并行执行，有依赖的任务由程序编排。
- 关键结果必须能在完全关闭 LLM 时由规则引擎完成。
- 人物 Agent 只在玩家召见或该人物被事件涉及时激活，不让 68 人每回合空转。

### 6.2 Agent 分工

| Agent | 输入 | 输出 | 是否影响数值 |
|---|---|---|---|
| 人物代理 | 人物知识、立场、当前议题 | 奏对、异议、命令草案 | 否 |
| 军事参谋 | 可见军情、军队与地形 | 2-4 个作战方案及风险 | 否 |
| 情报分析 | 多源报告和可信度 | 冲突点、缺口、建议复核项 | 否 |
| 政务参谋 | 财政、派系、民生 | 政治代价与执行风险 | 否 |
| 对手意图代理 | 敌方真实状态的受限视图 | 敌方行动候选 | 只提交候选 |
| 历史导演 | 历史锚点与当前因果链 | 事件候选、史实偏离说明 | 只提交候选 |
| 叙事代理 | 已结算的结构化结果 | 露布、朝报、人物反应 | 否 |
| 审校代理 | 文本、史料标签、结果对象 | 矛盾与越权报告 | 否 |

### 6.3 多 AI 一回合的并行方式

```text
                     ┌─ 军事参谋 ─┐
玩家命令 → 快照冻结 ├─ 情报分析 ─┤
                     ├─ 政务参谋 ─┼→ 候选集合 → 规则校验/冲突消解
                     ├─ 对手意图 ─┤                 ↓
                     └─ 历史导演 ─┘          确定性模拟与落库
                                                     ↓
                                  ┌─ 露布叙事 ───────┤
                                  ├─ 关键人物反应 ───┤ 并行、可流式
                                  └─ 审校与史料标签 ─┘
```

“多 AI”真正带来的收益应是不同角度、并行速度和有限职责，而不是把一次调用拆成更多收费调用。

### 6.4 冲突消解

Agent 输出永远是候选动作。程序按照以下顺序处理：

1. schema 与枚举合法性。
2. 角色权限与知识边界。
3. 资源、时间和地图可达性。
4. 与玩家已确认命令的冲突。
5. 同优先级动作的规则仲裁。
6. 随机种子驱动的有限不确定性。
7. 生成不可变的 `TurnResult`。

审校 Agent 不能覆盖 `TurnResult`，只可以标记文本与结果不一致并触发模板降级。

## 7. 回合提速方案

### 7.1 当前参考架构的主要瓶颈

- 一篇长叙事由高级模型生成后，又被财政、军队、人事和建筑抽取器重复读取。
- 多个高级模型调用串行执行，玩家必须等待全部完成才看到数值变化。
- LLM 输出格式漂移后依赖大量清洗，既慢又可能污染存档。
- 每回合把大块完整盘面重复塞给模型，token 随游戏进展增长。
- 叙事生成、规则计算和数据库写入没有清晰的快慢路径。

### 7.2 新回合流水线

| 阶段 | 目标预算 | 方法 |
|---|---:|---|
| 命令解析 | 0-1.5 秒 | 本地 schema 优先；自然语言解析仅一次小模型调用 |
| 快照与规则预计算 | <100 毫秒 | 脏域计算、内存快照、预编译规则 |
| AI 候选并发 | P50 <3 秒，P95 <8 秒 | 仅启动相关 Agent，统一上下文摘要，并发请求 |
| 确定性结算 | <300 毫秒 | 纯函数规则、单事务提交 |
| 首屏结果 | 结算后 <100 毫秒 | 立即返回结构化差异，不等叙事 |
| 叙事流 | 首字 <2 秒，完整 P95 <15 秒 | 结果落库后并行生成，可取消、可降级 |

硬性体验指标：

- 关闭 LLM 时，完整回合结算应小于 500 毫秒。
- 正常联网时，玩家在 P95 8 秒内看到可操作的结构化结果。
- 任一 Agent 超时不得阻塞存档或破坏回合；使用规则默认值或模板文本降级。
- 每回合高级模型调用默认不超过 1 次；其余优先小模型、缓存或程序规则。

### 7.3 具体优化手段

1. **一次结构化结算。** 先算数值，再写故事，不再从故事反推数值。
2. **脏域调度。** 没有军队行动就不运行战役 Agent；没有外交命令就不运行外部势力推演。
3. **上下文切片。** Agent 只收到与任务相关的地区、人物、军队和近因，不发送全库快照。
4. **并发 fan-out。** 军事、情报、政务和对手意图在只读快照上并行。
5. **结果缓存。** 静态人物设定、史料摘要、地图路径和 prompt 前缀使用内容哈希缓存。
6. **分层模型路由。** 规则与模板能完成的不调模型；人物短答用快速模型；关键章节才用高级模型。
7. **异步叙事。** UI 先显示“兵力 -3200、士气 -12、潼关仍守住”等事实，再逐步补充露布。
8. **摘要有上限。** 每个战役、人物和议题维护固定大小的结构化记忆，不把全部聊天历史滚入 prompt。
9. **单事务写入。** 一个回合的状态变化、事件和审计日志在同一数据库事务提交。
10. **可取消与幂等。** Agent 请求带 `turn_id + snapshot_hash`，重复响应不得重复落库。

### 8.3 状态与存储原则

- `WorldState` 是某一时点的完整权威状态。
- `Command` 是已确认输入，`TurnResult` 是不可变输出。
- 保存“命令日志 + 周期快照”，不必为追求概念完整而实现复杂全事件溯源。
- 每 5 回合或关键事件保存完整快照；其余回合保存差异和命令，可重放恢复。
- SQLite 开启 WAL；所有写操作经 repository 和单事务工作单元，不允许 Agent 工具直接 SQL。
- pending decision、Agent run、随机种子和上下文哈希全部落库，进程重启可恢复。

### 8.4 关键数据表

| 表 | 作用 |
|---|---|
| campaigns / snapshots | 战役与周期状态快照 |
| turns / commands / turn_results | 回合输入、输出与重放链 |
| regions / routes / armies / characters | 核心世界对象 |
| relationships / knowledge_claims | 关系网络与带来源情报 |
| events / event_instances | 事件定义与本局实例 |
| obligations | 回纥债务、赏赐、军饷和政治承诺 |
| agent_runs | 模型、耗时、token、schema、错误与缓存命中 |
| narratives | 可再生成的露布、朝报和对话文本 |
| source_citations | 事件、人物、数值与史料依据 |

### 8.5 API 边界

- `/campaigns/*`：新局、读档、快照、重放。
- `/court/*`：人物列表、召见、朝议、命令草案。
- `/turns/*`：校验命令、提交、读取结构化结果。
- `/streams/*`：人物回复、露布与后台审校流。
- `/codex/*`：史料与历史注释，不与运行时规则混在一起。
- API 文件按领域拆 router，禁止重新形成单文件巨石。

## 9. UI 与交互规划

### 9.1 第一屏

第一屏直接是可操作的危机桌面，不做营销式首页：

- 中央为关中—潼关—灵宝局部战略图。
- 左侧为当前危机、情报冲突和时间压力。
- 右侧为可召见人物与待处理奏报。
- 底部为本回合命令列、资源占用和“推演”按钮。
- 顶部只保留日期、皇统、威望、军费、粮运和全局警报。

### 9.2 关键工作流

1. 点击一条情报，查看来源、时间、支持与反证。
2. 从情报直接发起“召见相关人物”或“派遣复核”。
3. 在朝议中选择 2-5 名相关人物，不让全朝同时发言。
4. 把对话中的建议一键加入命令列，再编辑对象、资源和约束。
5. 推演前显示已知风险，不泄露隐藏真相。
6. 结算后先展示事实差异，再显示战报与人物反应。
7. 任意结果可展开“为何发生”，查看规则、命令和已知情报链。

### 9.3 地图层级

- 战略层：22 个地区、控制权、交通与补给。
- 战区层：军队、路线、粮仓、围城和战线。
- 专用战场层：仅灵宝、香积寺、相州等关键战役使用，不做通用战术沙盘。

## 10. 内容生产与史料治理

### 10.1 内容标签

每条内容必须标记：

- `[史实锚定]`：有明确史料依据的时间、人物或结果。
- `[史料争议]`：来源存在矛盾，并列主要说法。
- `[合理推演]`：符合制度与人物条件的游戏分支。
- `[虚构设计]`：为可玩性新增，不暗示为史实。

### 10.2 内容对象最小契约

人物必须有：身份、时间有效区间、地点、关系、知识边界、公开立场、私下动机、声音样例、史料来源。

事件必须有：候选窗口、程序 gate、前置因果、涉及对象、玩家可干预点、史实分支、替代分支、后果、来源与敏感内容等级。

数值必须有：单位、时间点、口径、来源等级、置信区间和游戏化变换说明。

### 10.3 内容优先级

MVP 先精做 16-20 名高频人物，而不是平均铺开 68 人：玄宗、肃宗、杨国忠、杨贵妃、高力士、边令诚、哥舒翰、王思礼、郭子仪、李光弼、李泌、安禄山、安庆绪、史思明、崔乾佑、陈玄礼、张巡、回纥使者等。

其余人物先有结构化档案和模板对话，等进入相关事件时再提升为完整人物代理。

## 11. 研发阶段与里程碑

以下用 **人周** 表示工作量，避免在团队规模未知时给出虚假日历承诺。按 4-6 人核心团队估算约 7-10 个月；单人全职更现实的范围是 14-20 个月。

### M0：架构与性能验证（8-12 人周）

交付：

- 新 `WorldState / Command / TurnResult` 契约。
- 无 LLM 的战斗、补给、忠诚与事件最小规则。
- 多 Agent 并发原型、超时降级、调用记录。
- 1000 回合 headless 基准与可重复随机种子。

验收：

- 相同输入和 seed 得到相同结果。
- 关闭 LLM 单回合 P95 <500 毫秒。
- 任一 Agent 超时不会破坏回合。
- 不存在从叙事文本反抽权威数值的路径。

### M1：潼关垂直切片（20-28 人周）

交付：

- 756 年六月危机回合。
- 6-8 名核心人物召见与朝议。
- 情报来源、复核、军令执行和灵宝专用战场。
- 长安守备/撤离/马嵬前置分支。
- 第一版地图、命令列、结果解释和露布流。

验收：

- 至少 5 种可解释结果，不是简单守/战二选一。
- 默认条件高概率走向史实惨败，但玩家能通过提前布局改变结果。
- 10 名外部测试者能在不看说明书的情况下完成一局。
- 正常联网首个可操作结果 P95 <8 秒。

### M2：第一至第三幕（28-40 人周）

交付：

- 玄宗出逃、马嵬、灵武、回纥谈判、睢阳与两京收复。
- 皇统切换、外援债务、围城和地区民生。
- 16-20 名完整人物代理，40+ 名结构化人物档案。
- 自动推进与剧情难度。

验收：

- 可连续游玩 3-5 小时无阻断。
- 关键债务能跨幕兑现，不能因换章丢失。
- 100 局自动模拟中无单一策略胜率超过 70%。

### M3：完整战役 Alpha（24-36 人周）

交付：

- 相州、史思明复叛、史朝义败亡、河朔安排、吐蕃压力。
- 完整终局评分与至少 8 个组合结局。
- 36-48 个有效回合的完整内容。
- 存档重放、崩溃恢复、历史注释模式。

验收：

- 500 局自动完整战役无死锁、负资源穿透或状态损坏。
- 任一关键事件可由规则解释其触发或未触发原因。
- 从任意关键快照重放可得到同一结构化结果。

### M4：Beta 与平衡（20-30 人周）

交付：

- UI 完整化、无障碍和性能优化。
- 史料审校、敏感内容分级、文本降级包。
- 难度曲线、教程、成就、Steam 基础接入。
- 100-300 名封测玩家数据。

验收：

- 崩溃率、调用失败率、平均回合时延和 token 成本达到预算。
- 主要路线都有明确代价，不存在无脑最优解。
- 历史顾问完成关键人物、事件和数值抽查。

### M5：发售候选（8-12 人周）

交付：

- 内容冻结、存档兼容策略、离线降级、用户隐私说明。
- 商店素材、演示版、日志脱敏和客服诊断包。
- 首日补丁与三个月更新计划。

## 13. 测试与质量门槛

### 13.1 测试金字塔

- 规则单测：每个公式、边界、状态迁移和权限。
- 属性测试：资源不凭空产生、兵力不为负、地图移动必须可达、死亡人物不能发令。
- 历史场景测试：默认潼关条件应稳定产生高惨败风险，但不是强制结果。
- 契约测试：所有 Agent 输出都通过 schema；失败走明确降级。
- 重放测试：命令、快照、seed 能复现结果。
- 负载测试：并发 Agent、长战役、自动推进和频繁存档。
- 内容测试：ID、日期、关系、来源、事件图和地图连通性。
- 前端流程测试：召见、组令、结算、中断、恢复和读档。

### 13.2 自动模拟

每个版本至少执行：

- 1000 次潼关开局蒙特卡洛。
- 500 次完整战役自动推演。
- 100 次故障注入：模型超时、JSON 错误、断网、进程重启、重复响应。
- 不同难度和关键策略的胜率、死亡率、藩镇债务、民生损失分布。

### 13.3 AI 质量评测

每类 Agent 建立固定评测集，关注：

- 人物是否越过知识边界。
- 是否把合理推演说成史实。
- 是否违背已结算事实。
- 建议是否重复、空泛或泄露隐藏信息。
- schema 成功率、首 token 延迟、总耗时、token 和缓存命中率。

## 14. 运营指标与性能预算

### 14.1 产品指标

- 垂直切片完成率。
- 玩家在首局识破或质疑假情报的比例。
- 每回合有效决策数与撤销率。
- 各路线选择分布，不追求平均，但要避免单一路线垄断。
- 玩家查看“为何发生”和史料注释的比例。
- 失败后继续率，而非只看胜率。

### 14.2 技术指标

- 单回合规则耗时 P50/P95。
- 首个可操作结果和完整叙事 P50/P95。
- 每回合 Agent 数、高级模型调用数与 token 成本。
- schema 失败率、超时率、降级率和缓存命中率。
- 存档恢复成功率与重放一致率。
- 100 回合内存增长与数据库体积。

建议建立开发期硬预算：**平均每有效回合模型成本不超过可接受售价的 0.1%-0.2%**，否则长局的单位经济性不可控。具体金额待模型供应商和商业模式确定后填写。

## 15. 主要风险与对策

| 风险 | 概率/影响 | 对策 |
|---|---|---|
| 系统过多导致永远做不完 | 高/高 | M1 只证明潼关；每个系统必须服务一个具体决策 |
| 多 AI 变成更多延迟和成本 | 高/高 | 脏域调度、并发、每回合调用上限、模板降级 |
| LLM 输出污染数值 | 中/高 | Agent 无写权限，所有变化经命令 schema 与 reducer |
| 历史结果与玩家自由冲突 | 高/高 | 历史提供压力和默认因果，不强制播放结果 |
| 兵力与史料数字争议 | 高/中 | 保存来源、口径和置信区间，展示有效战力解释 |
| 角色对话同质化 | 高/中 | 知识边界、利益、关系和声音样例共同约束 |
| 叙事与结算矛盾 | 中/高 | 叙事只读 `TurnResult`，审校失败则用模板 |
| 存档升级困难 | 中/高 | 版本化 schema、迁移测试、命令日志和周期快照 |
| 敏感历史与民族叙事 | 中/高 | 顾问审校、具体人物与政治结构叙事、避免族群本质化 |
| 单人或小团队资源不足 | 高/高 | 首发缩为短战役，后续章节扩展，不承诺双视角 |

## 16. 立项后前两周行动清单

### 第 1 周：冻结契约

1. 确认首发范围是完整 756-763，还是先发“潼关篇”。
2. 定义 `WorldState / Command / TurnResult` 最小 schema。
3. 把灵宝战役写成不依赖 LLM 的规则测试场景。
4. 建立回合 benchmark：规则耗时、AI 等待、token 和落库耗时。
5. 审计 D1-D7：人物数量、军队口径、地图连通、事件日期、来源等级。

### 第 2 周：验证新架构

1. 完成无 UI 的潼关 headless 原型。
2. 同时运行军事、情报、政务三个只读 Agent，输出统一 schema。
3. 用程序合并候选并生成唯一 `TurnResult`。
4. 结果返回后再异步生成一篇露布，验证不阻塞结算。
5. 连跑 1000 局，确认史实惨败能被系统自然解释，也存在高代价改写路线。

两周结束时只做一个去留判断：**新架构是否同时改善可解释性、回合时延和可测试性。** 如果没有，不进入大规模内容制作。

## 17. 尚需制作人确认的决策

这些问题不阻塞首稿，但必须在 M0 结束前定案：

1. 首次对外版本是完整八年战役，还是 756 年“潼关篇”章节版。
2. 玩家身份是固定皇帝、抽象王朝意志，还是允许玄宗/肃宗切换。

4. LLM 成本由开发者承担、玩家配置 API，还是混合方案。
5. 美术方向是写实历史、工笔绘卷，还是偏桌面军政档案。
6. 敏感内容的目标分级与发行地区。
7. 首发是否只做中文；若计划国际化，史料注释和人物称谓必须从第一天支持本地化键。

## 19. 最终建议

本项目最有价值的部分，不是把更多历史人物接到更多 Agent 上，而是把“为什么历史会这样发生”变成玩家可观察、可干预、可承担代价的系统。潼关之败尤其适合作为架构试金石：如果规则能在不写死结局的情况下自然产生史实风险，多 AI 能提供真正不同的有限视角，而玩家能在数秒内看见可解释结果，那么这套框架才值得扩展到完整安史之乱。

首个开发目标应当是一个快、稳、可重放的潼关危机，而不是先搬完 68 个人物、15 个事件和八年内容。
